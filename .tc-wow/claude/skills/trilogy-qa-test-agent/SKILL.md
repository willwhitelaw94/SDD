# Trilogy QA Test Agent

Autonomous browser testing agent spawned as a **Claude Code subagent** via the `Task` tool. Takes the test plan from `/trilogy-qa` (`qa-plan.md`), opens the browser, executes every test case, fixes failing code, re-verifies, and generates `test-report.md` with screenshot evidence. Runs in the background — go grab a coffee.

## Separation of Concerns

| Skill | Responsibility |
|-------|---------------|
| `/trilogy-qa` | **Plans** — generates qa-plan.md from spec ACs (no browser) |
| `/trilogy-qa-test-agent` | **Executes** — THIS: browser testing, fix failures, generate report |
| `/trilogy-qa-test-codify` | **Codifies** — converts passing browser tests into Pest/Playwright |

## When to Use

```bash
/trilogy-qa-test-agent                    # Execute test plan from qa-plan.md
/trilogy-qa-test-agent --plan path        # Point to specific test plan
/trilogy-qa-test-agent --report path      # Resume from existing test-report.md (re-test failures)
```

## Prerequisites

| Artifact | Required | Source |
|----------|----------|--------|
| qa-plan.md | Yes | Generated by `/trilogy-qa` — the test plan to execute |
| spec.md | Yes | Initiative folder — AC definitions for context |
| Code deployed locally | Yes | `npm run dev` / Herd serving the app |
| Chrome DevTools MCP | Yes | For browser testing |
| Laravel Boost MCP | Yes | For `get-absolute-url` |

## Execution Flow

### Phase 1 — Gather Context (main agent)

Before spawning the subagent, the main agent gathers all context needed:

1. **Find the test plan** by searching the epic folder:
   - Check `.tc-docs/content/initiatives/{initiative}/{epic}/qa-plan.md`
   - If no plan exists, tell the user to run `/trilogy-qa` first

2. **Read the test plan** — extract:
   - All test cases with steps, expected results, priority
   - Responsive test requirements
   - Accessibility checklist
   - Test data requirements

3. **Read spec.md** — get full AC definitions for context

4. **Get the app URL** using Laravel Boost `get-absolute-url` tool

5. **Get changed files** to understand scope:
   ```bash
   git diff --name-only origin/dev...HEAD
   ```

### Phase 2 — Spawn Subagent

6. **Launch the test agent** using the `Task` tool with `run_in_background: true`:

   ```
   Task(
     subagent_type: "general-purpose",
     description: "QA test execution agent",
     run_in_background: true,
     prompt: "<assembled prompt — see below>"
   )
   ```

7. **Assemble the prompt** — include ALL context the subagent needs:

   ```markdown
   You are a QA Test Agent. Your job is to execute the QA test plan in the
   browser, verify each test case, fix any failures, and generate a test report.

   ## Context

   **Feature**: [Feature Name]
   **App URL**: [url from get-absolute-url]
   **Test Plan**: [absolute path to qa-plan.md]
   **Spec**: [absolute path to spec.md]

   ## Test Plan

   [Paste the full qa-plan.md content]

   ## Changed Files (scope of investigation)

   [Paste git diff --name-only output]

   ## Your Protocol

   ### 0. Pre-Flight Setup — Database Self-Healing
   Before executing any test cases, ensure the database is ready:

   **Step 1: Check database state**
   ```sql
   SELECT COUNT(*) as total FROM users
   ```
   Use Laravel Boost `database-query` tool. If this query fails (table doesn't exist)
   or returns 0 users, the database needs restoring.

   **Step 2: Restore database (if empty)**
   Try snapshot restore first (~2s), fall back to full seed (~90s):
   ```bash
   # Try snapshot first
   php artisan db:snapshot restore --no-interaction
   ```
   If that fails (no snapshot file), fall back:
   ```bash
   # Full seed + auto-save snapshot for next time
   php artisan migrate:fresh --seed --no-interaction
   php artisan db:snapshot save --no-interaction
   ```

   **Step 3: If login fails mid-test** (redirected to login page unexpectedly,
   or Chrome shows login screen when navigating to a protected page):
   - Re-check `SELECT COUNT(*) as total FROM users` — if 0, repeat Step 2
   - If users exist but login fails, check password: the password is always `foobar`
   - Check the user has `has_login = 1` and `email_verified_at IS NOT NULL`

   **Step 4: Enable feature flags and verify test data**
   - Use Laravel Boost `tinker` tool to enable any feature flags listed in the test plan
   - Use Laravel Boost `database-query` tool to verify seeded test data exists
   - If the test plan has a "Setup Prerequisites" section, execute every step in it
   - Navigate to the feature's main page in Chrome DevTools to confirm access

   For EACH test case in the plan, in priority order (P1 first):

   ### 1. Execute
   - Navigate to the relevant page using Chrome DevTools MCP `navigate_page`
   - Use `take_snapshot` to read the page structure and find elements
   - Execute the test steps using `click`, `fill`, `fill_form`
   - Use `wait_for` to wait for expected outcomes
   - Use `take_screenshot` to capture evidence (save to .qa/screenshots/)
   - Check console with `list_console_messages` for errors

   ### 2. Record Result
   - PASS: Test case works as expected, screenshot captured
   - FAIL: Expected result not met, screenshot of failure captured
   - BLOCKED: Cannot execute (missing prerequisite, broken dependency)

   ### 3. Fix Failures (if FAIL)
   - Read the relevant source code
   - Make the minimal code change needed
   - Run relevant automated tests:
     ```bash
     php artisan test --compact --filter=[relevant]
     ```
   - Run code formatting:
     ```bash
     vendor/bin/pint --dirty
     ```
   - Re-verify in the browser
   - If fix works, mark as PASS with notes about what was fixed

   ### 4. Move to Next Test Case

   ## After All Test Cases

   1. Run responsive tests (desktop 1920x1080, tablet 768x1024, mobile 375x812)
      using Chrome DevTools `emulate` tool
   2. Run accessibility spot-checks using `take_snapshot` with verbose: true
   3. Run the full test suite:
      ```bash
      php artisan test --compact
      ```
   4. Run code formatting:
      ```bash
      vendor/bin/pint --dirty
      ```
   5. Generate test-report.md (see report template below)

   ## Test Report Template

   Generate test-report.md with:
   - Automated test results (total, passed, failed)
   - AC verification table (TC number, criteria, method, result, evidence)
   - Responsive testing results (3 breakpoints with screenshots)
   - Console & network audit
   - Accessibility spot-check results
   - Issues found (with severity)
   - QA Gate 5 readiness assessment

   ## Rules

   - Execute test cases in priority order (P1 first, then P2, then P3)
   - Always verify in the browser — automated tests alone aren't enough
   - Never mark a test as PASS without a screenshot as evidence
   - If stuck on a test after 3 attempts, mark BLOCKED with detailed notes
   - Fix ONE failure at a time — don't batch fixes
   - Run tests after every code fix to avoid regressions
   - Use `wait_for` liberally — Inertia pages load asynchronously
   - test-report.md is the deliverable — keep it updated throughout
   ```

### Phase 3 — Inform and Monitor

8. **Tell the user the agent is running**:

   > "Test agent spawned in the background. It will execute the test plan — testing each case in the browser, fixing failures, and generating the test report.
   >
   > Go grab a coffee, this will take a while.
   >
   > I'll let you know when it's done. You can also check progress by reading test-report.md."

9. **When the agent completes**, read the generated test-report.md and summarise:
   - How many test cases passed/failed/blocked
   - Any code fixes made
   - Whether responsive testing passed
   - Recommendation: ready for `/trilogy-qa-handover` or needs manual attention

## Chrome DevTools MCP Tools Reference

| Tool | Use For |
|------|---------|
| `navigate_page` | Go to pages, navigate flows |
| `take_snapshot` | Read page structure (a11y tree), find element UIDs |
| `take_screenshot` | Visual evidence of state |
| `click` | Click buttons, links, tabs |
| `fill` / `fill_form` | Enter form data |
| `wait_for` | Wait for text/elements to appear |
| `list_console_messages` | Check for JS errors/warnings |
| `list_network_requests` | Check for 404s, failed requests |
| `emulate` | Responsive testing (viewport, touch, dark mode) |
| `resize_page` | Change window size |
| `evaluate_script` | Run custom JS checks |
