# Gate 5: QA Gate

**Transition**: QA Testing → Release
**Linear Transition**: **QA** → **Release**
**meta.yaml**: *(updated by skill)*

**Key Question**: "Does it actually work?"

Verifies the feature is stable, meets acceptance criteria, and is ready for release. Run after QA testing is complete, via `/trilogy-qa-handover`.

---

## Prerequisites

| Artifact | Required | Location |
|----------|----------|----------|
| Code Quality Gate (Gate 4) passed | Yes | PR/Linear |
| Feature deployed to staging | Yes | Staging environment |
| Test data available | Recommended | Developer notes |

---

## Checks

### 1. Functional Validation

| Check | Pass Criteria |
|-------|---------------|
| **Feature works as described** | New feature matches ticket description |
| **Acceptance criteria met** | All AC from spec.md verified |
| **UI matches approved design** | Visual comparison to Figma/mockups |
| **Field validation works** | Required fields, format validation, error states |
| **Success messages display** | Confirmation feedback for user actions |
| **Error states handled** | Graceful error handling, helpful messages |

### 2. Playwright E2E Tests

| Check | Pass Criteria | How to Verify |
|-------|---------------|---------------|
| **E2E tests exist** | Page object + spec file in `e2e/` | `ls e2e/tests/{feature}/` |
| **E2E tests pass** | All Playwright tests green | `cd e2e && npx playwright test tests/{feature}/ --reporter=list` |
| **Happy path covered** | Main user flow tested end-to-end | Review spec file |
| **Validation covered** | Key validation rules have test cases | Review spec file |

### 3. Regression & Integration


| Check | Pass Criteria |
|-------|---------------|
| **Regression check passed** | Related areas unaffected by changes |
| **Link integrity confirmed** | All navigation links work correctly |
| **Data integrity maintained** | No data corruption or loss |

### 4. Cross-Browser & Device Testing

| Check | Pass Criteria |
|-------|---------------|
| **Chrome tested** | Desktop + mobile |
| **Safari tested** | Desktop + mobile |
| **Firefox tested** | Desktop + mobile |
| **Responsive on top 3 screen sizes** | Desktop (1920), Tablet (768), Mobile (375) |

### 5. Quality & Performance

| Check | Pass Criteria |
|-------|---------------|
| **Accessibility check** | WCAG 2.1 AA basic pass |
| **Performance acceptable** | No visible lag, no layout shift |
| **No console warnings or 404s** | Browser console clean |

### 6. Issue Management

| Check | Pass Criteria |
|-------|---------------|
| **No open Severity 1-3 issues** | Critical/major bugs resolved |
| **Minor issues documented** | Known issues accepted by Product Owner |

### 7. QA Sign-off

| Check | Pass Criteria |
|-------|---------------|
| **QA Tester confirms** | Ticket is stable and ready for Release |

---

## Severity Definitions

| Severity | Description | Gate Impact |
|----------|-------------|-------------|
| **Sev 1 - Critical** | Feature unusable, data loss, security issue | BLOCKS gate |
| **Sev 2 - Major** | Core functionality broken, no workaround | BLOCKS gate |
| **Sev 3 - Moderate** | Functionality impaired, workaround exists | BLOCKS gate |
| **Sev 4 - Minor** | Cosmetic, minor UX issue | Document & proceed |
| **Sev 5 - Trivial** | Edge case, future enhancement | Document & proceed |

---

## Gate Actions

### On Pass
- Move ticket to "Ready for Release"
- Add "QA Approved" label
- Tag Product Owner for review
- Log gate passage

### On Fail
- List failed checks with specific issues
- Create bug tickets for Sev 1-3 issues
- Return to developer for fixes
- Re-run gate after fixes
