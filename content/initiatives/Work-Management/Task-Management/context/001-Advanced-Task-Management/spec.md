## User Scenarios & Testing

### User Story 1 - View and Manage Tasks in Kanban Board (Priority: P1)

As an internal user or coordinator, I want to view my tasks in a Kanban board so that I can quickly see task status and drag tasks between columns to update their progress.

**Why this priority**: Kanban is the primary interaction model—users need a visual, interactive way to manage tasks. This is the core differentiator from v1.

**Independent Test**: Can be fully tested by creating tasks, viewing them on the Kanban board, and dragging between columns. Delivers immediate value for task visualization and status management.

**Acceptance Scenarios**:

1. **Given** I am logged in as an internal user, **When** I navigate to the task dashboard, **Then** I see a Kanban board with columns for each task status (To Do, In Progress, Done)
2. **Given** I have tasks assigned to me, **When** I view the Kanban board, **Then** I see my tasks displayed as cards in the appropriate status columns
3. **Given** I am viewing a task card, **When** I drag it to a different column, **Then** the task status updates immediately and persists
4. **Given** I am a coordinator, **When** I view the Kanban board, **Then** I see tasks for my assigned scope (not all tasks system-wide)

---

### User Story 2 - Filter and Sort Tasks in Table View (Priority: P1)

As an internal user or coordinator, I want to view tasks in a table format with filtering and sorting so that I can find specific tasks quickly and manage large task lists.

**Why this priority**: Table view complements Kanban for users who prefer list-based workflows. Reuses existing table components.

**Independent Test**: Can be fully tested by viewing task table, applying filters (status, assignee, due date), and sorting columns. Delivers immediate value for task discovery.

**Acceptance Scenarios**:

1. **Given** I am logged in, **When** I switch to table view, **Then** I see all my tasks in a sortable, filterable table
2. **Given** I am viewing the task table, **When** I filter by status "In Progress", **Then** only tasks with that status are displayed
3. **Given** I am viewing the task table, **When** I click a column header, **Then** tasks are sorted by that column (ascending/descending toggle)
4. **Given** I have many tasks, **When** I apply multiple filters (status + due date + assignee), **Then** results update in real-time

---

### User Story 3 - Create Tasks Manually (Priority: P1)

As an internal user or coordinator, I want to create new tasks manually so that I can track work items that aren't auto-generated by the system.

**Why this priority**: Users must be able to create their own tasks—this is fundamental task management functionality.

**Independent Test**: Can be fully tested by opening task creation form, filling required fields, and saving. Delivers immediate value for task capture.

**Acceptance Scenarios**:

1. **Given** I am logged in, **When** I click "New Task", **Then** I see a task creation form with required fields (title, description, due date, assignee)
2. **Given** I am creating a task, **When** I fill required fields and save, **Then** the task appears in my task list and Kanban board
3. **Given** I am creating a task, **When** I link it to a business object (patient, package, etc.), **Then** the task shows the relationship in task details
4. **Given** I leave required fields empty, **When** I try to save, **Then** I see validation errors indicating missing fields

---

### User Story 4 - Accept AI-Recommended Tasks (Priority: PHASE 2 - DEFERRED)

> **DEFERRED TO PHASE 2**: AI recommendations require separate planning effort. Ship core task management first.

As an internal user or coordinator, I want to see AI-recommended tasks when viewing business objects (patients, packages, risks, etc.) so that I can quickly capture relevant tasks without manually creating them.

**Why deferred**: AI recommendations add significant complexity (rules engine, per-module triggers, testing). Core Kanban/table/keyboard features deliver immediate value and can ship by March. AI can follow once adoption baseline is established.

**Acceptance Scenarios** (for Phase 2):

1. **Given** I am viewing a patient record, **When** the system identifies recommended tasks, **Then** I see a "Recommended Tasks" section with suggested tasks
2. **Given** I see a recommended task, **When** I click "Accept", **Then** the task is created and linked to that patient
3. **Given** I see a recommended task, **When** I click "Dismiss", **Then** the recommendation is hidden (but may reappear if conditions persist)
4. **Given** I am viewing a package, **When** AI detects overdue actions, **Then** I see relevant task recommendations specific to that package

---

### User Story 5 - View Tasks Linked to Business Objects (Priority: P2)

As an internal user or coordinator, I want to see all tasks related to a specific patient, package, or other business object so that I can understand outstanding work in context.

**Why this priority**: Cross-module visibility is critical for contextual task management—users need to see tasks where they work.

**Independent Test**: Can be fully tested by navigating to a patient/package and viewing the "Related Tasks" section. Delivers immediate value for context.

**Acceptance Scenarios**:

1. **Given** I am viewing a patient record, **When** tasks exist for that patient, **Then** I see a "Related Tasks" panel showing linked tasks
2. **Given** I am viewing a package, **When** I click on a related task, **Then** I see task details without leaving the package view
3. **Given** a task is linked to multiple objects (patient + package), **When** I view either object, **Then** the task appears in both "Related Tasks" panels

---

### User Story 6 - Assign Tasks to Users (Priority: P2)

As a coordinator, I want to assign tasks to internal users so that work is distributed and tracked across the team.

**Why this priority**: Task assignment enables team coordination—essential for coordinator workflows.

**Independent Test**: Can be fully tested by creating a task, selecting an assignee, and verifying the assignee sees the task. Delivers immediate value for team coordination.

**Acceptance Scenarios**:

1. **Given** I am creating or editing a task, **When** I select an assignee, **Then** the task appears in that user's task list
2. **Given** I am a coordinator, **When** I assign a task to an internal user, **Then** they see the task in their Kanban board and table view
3. **Given** a task is assigned to me, **When** I complete it, **Then** the coordinator sees the updated status

---

### User Story 7 - Care Partner Task Visibility (Priority: P3)

As a care partner, I want to see tasks related to my patients so that I understand coordination activities and outstanding work.

**Why this priority**: Care partner visibility is valuable but not core to internal workflows—can be delivered after P1/P2.

**Independent Test**: Can be fully tested by logging in as care partner, navigating to patient, and viewing related tasks. Delivers value for external stakeholder engagement.

**Acceptance Scenarios**:

1. **Given** I am logged in as a care partner, **When** I view a patient I'm associated with, **Then** I see a read-only view of related tasks
2. **Given** I am a care partner viewing tasks, **When** I see task details, **Then** I cannot edit or reassign (view-only access)
3. **Given** a task is marked sensitive, **When** I view related tasks, **Then** sensitive tasks are hidden from care partner view

---

### User Story 8 - Task Due Date and Reminders (Priority: P3)

As an internal user or coordinator, I want tasks to have due dates and visual indicators so that I can prioritize urgent work.

**Why this priority**: Due dates enhance task management but are not required for core functionality.

**Independent Test**: Can be fully tested by creating tasks with due dates and verifying visual indicators. Delivers value for prioritization.

**Acceptance Scenarios**:

1. **Given** I create a task with a due date, **When** I view the Kanban board, **Then** the task card shows the due date
2. **Given** a task is overdue, **When** I view the Kanban board or table, **Then** the task is visually highlighted (e.g., red border)
3. **Given** a task is due within 24 hours, **When** I view it, **Then** I see an "urgent" indicator

---

### User Story 9 - Keyboard Shortcuts & Command Palette (Priority: P2)

As a power user (internal user or coordinator), I want keyboard shortcuts and a command palette so that I can manage tasks quickly without using the mouse.

**Why this priority**: Linear-style keyboard-first UX dramatically improves power user efficiency. Key differentiator for adoption among frequent users.

**Independent Test**: Can be fully tested by pressing keyboard shortcuts and verifying actions execute. Delivers immediate value for speed.

**Acceptance Scenarios**:

1. **Given** I am on the task dashboard, **When** I press `Cmd/Ctrl+K`, **Then** a command palette opens with search and quick actions
2. **Given** the command palette is open, **When** I type "new task", **Then** I can create a task without clicking
3. **Given** I am viewing a task, **When** I press `E`, **Then** I can quickly edit the task
4. **Given** I am on the Kanban board, **When** I press `1/2/3`, **Then** the selected task moves to column 1/2/3
5. **Given** I am viewing tasks, **When** I press `/`, **Then** the filter search activates instantly
6. **Given** I press `?`, **When** I am anywhere in the task system, **Then** I see a keyboard shortcuts help overlay

**Core Shortcuts** (Linear-inspired):
- `Cmd/Ctrl+K` — Command palette
- `C` — Create new task
- `E` — Edit selected task
- `/` — Focus filter/search
- `1-9` — Move task to status column
- `A` — Assign task
- `D` — Set due date
- `?` — Show shortcuts help

---

### User Story 10 - Subtasks (Priority: P2)

As an internal user or coordinator, I want to break down tasks into subtasks so that I can track granular progress on complex work items.

**Why this priority**: Subtasks are standard in Asana/Linear and critical for tracking multi-step work. Adds depth without complexity.

**Independent Test**: Can be fully tested by creating a parent task, adding subtasks, and marking them complete. Delivers value for granular tracking.

**Acceptance Scenarios**:

1. **Given** I am viewing a task, **When** I click "Add Subtask", **Then** I can create a subtask linked to the parent
2. **Given** a task has subtasks, **When** I view the task card, **Then** I see a subtask progress indicator (e.g., 2/5 complete)
3. **Given** I complete all subtasks, **When** I view the parent task, **Then** I see a prompt to mark the parent complete
4. **Given** I am on the Kanban board, **When** I view a task with subtasks, **Then** the card shows subtask count

---

### User Story 11 - Task Comments & Activity (Priority: P2)

As an internal user or coordinator, I want to add comments to tasks and see activity history so that I can collaborate with my team without leaving the task.

**Why this priority**: Comments enable async collaboration—essential for coordinator handoffs and team coordination.

**Independent Test**: Can be fully tested by adding comments to a task and viewing activity log. Delivers value for team collaboration.

**Acceptance Scenarios**:

1. **Given** I am viewing a task, **When** I add a comment, **Then** the comment appears in the task's activity feed
2. **Given** I mention a user with @username, **When** I save the comment, **Then** that user is notified
3. **Given** someone changes task status or assignee, **When** I view the task, **Then** I see the change in the activity log with timestamp
4. **Given** I am a care partner viewing a task, **When** I see comments, **Then** I can view but not add comments (unless configured otherwise)

### Edge Cases

- What happens when a user tries to assign a task to themselves? → Allowed, task appears in their list
- What happens when a linked business object is deleted? → Task remains but shows "Object deleted" indicator
- How does the system handle duplicate AI recommendations? → Same recommendation not shown twice within 24 hours
- What happens when a coordinator tries to view tasks outside their scope? → Tasks are hidden based on role permissions
- What if a user has 500+ tasks? → Pagination and performance optimization required; default view shows last 30 days

## Requirements

### Functional Requirements

**Core Task Management**
- **FR-001**: System MUST display tasks in a Kanban board view with configurable status columns
- **FR-002**: System MUST allow users to drag-and-drop tasks between Kanban columns to update status
- **FR-003**: System MUST display tasks in a table view with column sorting (ascending/descending)
- **FR-004**: System MUST allow users to filter tasks by status, assignee, due date, and linked object type
- **FR-005**: System MUST allow users to create tasks with title, description, due date, and assignee fields
- **FR-006**: System MUST persist task data including status changes, assignments, and linked objects

**AI Recommendations (PHASE 2 - DEFERRED)**
- **FR-007**: _(Phase 2)_ System MUST display AI-recommended tasks contextually on business object views
- **FR-008**: _(Phase 2)_ System MUST allow users to accept AI-recommended tasks with one click
- **FR-009**: _(Phase 2)_ System MUST allow users to dismiss AI-recommended tasks
- **FR-010**: _(Phase 2)_ System MUST link accepted tasks to the originating business object automatically

**Task Relationships**
- **FR-011**: System MUST allow tasks to be linked to one or more business objects (patients, packages, risks, bills, supplies, coordinators)
- **FR-012**: System MUST display related tasks on business object detail views
- **FR-013**: System MUST maintain task links even if the linked object's data changes

**User Roles & Permissions**
- **FR-014**: System MUST restrict task visibility based on user role (internal user, coordinator, care partner)
- **FR-015**: System MUST allow coordinators to assign tasks to internal users
- **FR-016**: System MUST provide care partners with read-only access to relevant tasks
- **FR-017**: System MUST hide sensitive tasks from care partner view

**Due Dates & Visual Indicators**
- **FR-018**: System MUST display due dates on task cards and table rows
- **FR-019**: System MUST visually highlight overdue tasks
- **FR-020**: System MUST visually indicate tasks due within 24 hours

**Keyboard Shortcuts & Command Palette**
- **FR-021**: System MUST provide a command palette accessible via `Cmd/Ctrl+K`
- **FR-022**: System MUST support keyboard shortcuts for common actions (create, edit, assign, move status)
- **FR-023**: System MUST display a keyboard shortcuts help overlay when user presses `?`
- **FR-024**: System MUST allow users to navigate and select tasks using arrow keys
- **FR-025**: System MUST support quick filtering via `/` keyboard shortcut

**Subtasks**
- **FR-026**: System MUST allow users to create subtasks within a parent task
- **FR-027**: System MUST display subtask progress on parent task cards (e.g., 2/5 complete)
- **FR-028**: System MUST allow subtasks to be marked complete independently
- **FR-029**: System MUST prompt user to complete parent task when all subtasks are done

**Comments & Activity**
- **FR-030**: System MUST allow users to add comments to tasks
- **FR-031**: System MUST support @mentions in comments to notify other users
- **FR-032**: System MUST display an activity log showing all task changes with timestamps
- **FR-033**: System MUST allow care partners to view comments but not add them (default)

### Key Entities

- **Task**: Core work item with title, description, status, due date, assignee, linked objects, parent task (for subtasks), created/updated timestamps
- **Subtask**: Child task linked to a parent task, inherits parent's linked objects
- **Task Status**: Configurable workflow states (e.g., To Do, In Progress, Done)
- **Task Link**: Relationship between a task and a business object (polymorphic: patient, package, risk, bill, supply, coordinator)
- **Task Comment**: User comment on a task with @mentions support
- **Task Activity**: Audit log entry for task changes (status, assignee, etc.)
- **AI Recommendation**: Suggested task generated by rules engine, linked to triggering business object
- **User Role**: Permission level determining task visibility and actions (internal user, coordinator, care partner)

## Success Criteria

### Measurable Outcomes

- **SC-001**: 60% of active users create or complete at least one task per week within 3 months of launch
- **SC-002**: Average user manages 5+ tasks per week (creation + completion)
- **SC-003**: 70% of created tasks are marked complete within their SLA/due date
- **SC-004**: _(Phase 2)_ 40% of AI-recommended tasks are accepted by users
- **SC-005**: Users can create a new task in under 30 seconds
- **SC-006**: Kanban drag-and-drop status update completes in under 1 second
- **SC-007**: Table view filters apply in under 2 seconds for users with up to 500 tasks
- **SC-008**: Care partners can view related tasks within 3 clicks from patient view
