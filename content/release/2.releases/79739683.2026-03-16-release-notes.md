## TL;DR

This release lays the groundwork for the Budgets Reloaded project with new V2 budget pages and a funding streams API, while also introducing a mobile notification system for recipients. Billing gains a fee exemption audit trail so organisations can track who exempted loading fees, and supplier compliance logic has been corrected to properly recognise police check and NDIS worker screening documents. Performance has been significantly improved through database index optimisations and cursor-based pagination for notifications.

## What's New?

:::tab{label="Business View"}

### Budgets Reloaded — New budget management system foundations

**Why this matters**

The current budget management workflow requires significant manual effort and lacks visibility into funding stream breakdowns. Budgets Reloaded is the strategic initiative to modernise how budgets are created, viewed, and managed across packages.

**What's changing**

The first phase of the new budget system has been built, introducing a redesigned budget page with improved plan selection, grouped budget plans, and a dedicated funding streams view. This release establishes the backend APIs and frontend structure that all future budget features will build upon.

**Who's affected**

Care Coordinators, Finance team, and anyone involved in package budget management.

**Action required**

No action required at this stage — these are foundational changes. The new budget interface will be progressively rolled out as additional capabilities are completed.

---

### Fee exemption audit trail — Track who exempted loading fees on bills

**Why this matters**

Previously, when loading fees were exempted on a bill, there was no record of who made that decision. This creates compliance and accountability gaps, particularly during audits.

**What's changing**

The system now records which user exempted loading fees on each bill. This information is visible as a new column in the Bills table, providing a clear audit trail for fee exemption decisions.

**Who's affected**

Finance team and billing staff who manage fee exemptions.

**Action required**

No action required — this is available immediately. Existing exemptions will be backfilled where the data is recoverable.

---

### Supplier compliance verification improvements

**Why this matters**

Supplier verification logic was not correctly recognising police check and NDIS worker screening documents, potentially blocking compliant suppliers from progressing through onboarding. Separately, expired document alerts were appearing incorrectly on supplier dashboards.

**What's changing**

The verification process now correctly includes police check and NDIS worker screening in its compliance assessment. Expired document alerts have been refined to show accurately based on document status. Business verification documents are now visible regardless of whether a supplier has services in the price list.

**Who's affected**

Compliance team, supplier onboarding coordinators, and suppliers themselves.

**Action required**

No action required — these corrections are applied automatically.

---

### New compliance document permission

**Why this matters**

Compliance team members needed the ability to add missing documents on behalf of suppliers to unblock verification workflows.

**What's changing**

A new permission allows Admin, Compliance Manager, and Compliance Team Member roles to add missing supplier documents directly.

**Who's affected**

Compliance team.

**Action required**

No action required — the permission has been added to the relevant roles automatically.

---

### Mobile recipient notification system

**Why this matters**

Recipients using the mobile app had no way to receive or manage in-app notifications, limiting the platform's ability to communicate time-sensitive updates to care recipients.

**What's changing**

A new notification system has been built for the mobile experience, allowing recipients to view, manage, and mark notifications as read directly from the mobile app.

**Who's affected**

Recipients using the mobile application, and teams responsible for recipient communications.

**Action required**

No action required — this is available to mobile app users immediately.

---

### Team permission system fix

**Why this matters**

Permission checks were not correctly scoping to teams, which could result in users seeing incorrect access levels or being unable to perform actions they should have access to.

**What's changing**

The permission system's team context handling has been corrected, ensuring permissions are properly evaluated within the correct team scope.

**Who's affected**

All portal users, particularly those in multi-team organisations.

**Action required**

No action required — this fix is applied automatically.

---

### Support at Home is now always enabled

**Why this matters**

The Support at Home (SAH) feature flag was a legacy toggle from the initial rollout. All organisations now operate under Support at Home, making the toggle unnecessary overhead.

**What's changing**

The SAH_ENABLED configuration flag has been removed. All Support at Home functionality is now permanently active across all organisations, simplifying the system and eliminating configuration drift risk.

**Who's affected**

No impact on users — this formalises what was already the operational reality.

**Action required**

No action required.

---

### Platform performance improvements

**Why this matters**

Several high-traffic database queries were running without optimal indexes, causing slower page loads particularly for users with many notifications and for supplier profile lookups.

**What's changing**

Database indexes have been added to key tables (notifications, organisation workers, verification logs), notification loading now uses cursor-based pagination for faster scrolling, and settings lookups are cached in-memory to reduce repeated database calls.

**Who's affected**

All portal users will experience faster page loads.

**Action required**

No action required.

### You can now see why a bill was put on hold or rejected

**Why we built this**

Previously, when a bill was placed on hold or rejected, you had no visibility into the reason — you'd have to contact support or check elsewhere to understand what happened.

### Bill items with linked transactions can no longer be accidentally deleted

**Why we built this**

Previously, it was possible to delete a bill item even if it had linked financial transactions, which could cause data integrity issues.

**What's changed**

The system now prevents deletion of bill items that have linked transactions and shows a clear error message explaining why the item cannot be removed.

**Where to find it**

`Bills → Edit a bill → Attempt to delete a line item`

**How to use it**

If you try to delete a bill item with linked transactions, you'll see a message explaining that the item cannot be deleted. You'll need to resolve the linked transactions first.

### Bill on-hold/rejection reasons now visible to users

**Why this was changed**

Users were contacting support to ask why their bills were on hold or rejected, because the reason wasn't visible in the portal.

**What's different**

Bills now display the on-hold or rejection reason directly on the bill detail page. Both Care Coordinator and Supplier views show this information.

**Where in the platform**

`Bills → Select any on-hold or rejected bill` — the reason appears near the status badge.

**How to help users**

If a user asks about a bill on hold, direct them to the bill detail page where the reason is now displayed. This should reduce the volume of "why is my bill on hold?" queries.

**Likely Q&A**

> **Q: "I can see the bill is on hold but there's no reason shown?"**
> A: The reason field is only populated for bills placed on hold after this release. Older on-hold bills may not have the reason displayed.

> **Q: "The on-hold comment just says error details I don't understand?"**
> A: For Care Management bills, the on-hold comment now includes technical validation details. If the user needs help interpreting these, escalate to the billing team.

### Fee exemption tracking — new "Exempted By" column on Bills

**Why this was changed**

There was no audit trail for who exempted loading fees on bills, making it difficult to investigate fee exemption decisions.

**What's different**

A new "Exempted By" column appears in the Bills table showing which user exempted the loading fees. The system also tracks new fee exemption filter options.

**Where in the platform**

`Bills → Bills table` — new columns for fee exemption details.

**How to help users**

If a user asks who exempted fees on a bill, direct them to the fee exemption columns in the Bills table.

**Likely Q&A**

> **Q: "The 'Exempted By' column is empty for some bills?"**
> A: The tracking was just introduced. Older exemptions are being backfilled where possible, but some historical data may not be available.

### Supplier verification logic updated

**Why this was changed**

Some suppliers were stuck in onboarding because the system wasn't correctly recognising police check or NDIS worker screening documents as valid verification documents.

**What's different**

The verification check now correctly includes police check and NDIS worker screening documents. Suppliers who were previously stuck should now progress through the verification stages.

**Where in the platform**

Affects the supplier verification status shown in `Suppliers → Dashboard` and the supplier document compliance views.

**How to help users**

If a supplier reports being stuck in verification despite having uploaded police check or NDIS screening documents, this fix should have resolved it. Ask them to refresh the page or log out and back in.

**Likely Q&A**

> **Q: "My supplier still shows as not verified even though they uploaded documents?"**
> A: Ask the supplier to check which specific documents they've uploaded. The fix covers police check and NDIS worker screening specifically. Other document types may still be required.

> **Q: "A supplier's expired document alerts seem wrong?"**
> A: The alert logic was also updated in this release. Ask the supplier to refresh the page. If the issue persists, escalate to the development team.

### New permission: Compliance can add missing supplier documents

**Why this was changed**

Compliance team members needed the ability to upload missing documents on behalf of suppliers to unblock the verification process.

**What's different**

Admin, Compliance Manager, and Compliance Team Member roles can now add missing documents to a supplier's record.

**Where in the platform**

`Suppliers → Staff view → Missing Documents tab`

**How to help users**

Deleting a bill item that had financial transactions linked to it could cause data integrity issues.

**What's different**

The system now blocks deletion of bill items that have linked transactions and shows an error message explaining why.

**Where in the platform**

`Bills → Edit bill → Delete a line item`
