## TL;DR

This release introduces a new containerised cloud infrastructure for the TC Portal, built on AWS Elastic Container Service (ECS) Fargate and managed with AWS CDK. The application is now packaged as a Docker image and deployed through an automated GitHub Actions CI/CD pipeline, replacing the previous deployment approach. All four core services — web, background worker, scheduler, and search — run as independent, auto-scaling Fargate containers with persistent storage and CloudWatch monitoring. This change has no visible impact on users or day-to-day operations.

---

## What's New?

::tabs{default="Business View"}
:::tab{label="Business View"}

### Cloud-Native Infrastructure Modernisation

The TC Portal is now deployed on AWS Elastic Container Service (ECS) Fargate — a serverless container platform that removes the need to manage individual servers.

**Why this matters**

Operating on managed cloud infrastructure reduces operational burden, improves deployment consistency, and provides a foundation for future scalability. By containerising the application, the business gains repeatable, auditable deployments and eliminates environment-specific configuration drift that can cause reliability issues.

**What's changing**

The application is now packaged as a Docker container and deployed to AWS using Infrastructure as Code (CDK). All application components — the web server, background job processor, scheduler, and search engine — run as separate, independently scalable services. A full automated CI/CD pipeline handles building and deploying new versions without manual intervention.

**Who's affected**

Engineering and DevOps teams. This is a backend infrastructure change with no operational impact on coordinators, recipients, suppliers, or any end user of the portal.

**Action required**

No action required — this is a backend infrastructure change and is completely transparent to all users and operational teams.

:::tab{label="Support View"}

### Infrastructure Modernisation — No User-Facing Impact

The TC Portal application has been containerised and moved to AWS ECS Fargate with an automated CI/CD deployment pipeline.

**Why this was changed**

The engineering team has modernised the application's deployment infrastructure to improve consistency, reliability, and scalability of the platform. This is a planned infrastructure investment.

**What's different**

Nothing has changed for users. All portal functionality remains identical. Deployments are now automated through a Docker-based CI/CD pipeline rather than manual processes.

**Where in the platform**

Not applicable — this is an infrastructure-level change with no visible footprint in the portal.

**How to help users**

If users report any unexpected behaviour after this release, escalate to engineering — but this change should be completely transparent. No new permissions, no new features, no workflow changes.

**Likely Q&A**

> **Q: "Will there be any downtime?"**
> A: Deployments using the new pipeline are designed to be zero-downtime rolling updates. If there is any planned maintenance window, engineering will communicate this separately.

  ### Change Log

**[#6238](https://github.com/Trilogy-Care/tc-portal/pull/6238) — feat: AWS CDK infrastructure for ECS Fargate deployment**

- Two-stack CDK pattern: `FoundationStack` + `ApplicationStack`
- `FoundationStack` (`infrastructure/lib/foundation-stack.ts`): VPC (2 AZs), RDS MySQL 8.0, ElastiCache Redis 7.x, S3
- `ApplicationStack` (`infrastructure/lib/application-stack.ts`): ECS cluster, ALB, 4 Fargate task definitions, auto-scaling, CloudWatch alarms + dashboard, Container Insights
- 4 ECS Fargate services: `web` (Nginx + PHP-FPM), `worker` (Laravel Horizon), `scheduler`, `typesense`
- EFS persistence for Typesense with IAM-authenticated access point
- Cross-account ECR image pull from DevOps account `967883357946` via STS `AssumeRole`
- Deployment target: sandbox account `935598634652` (staging only)
- CDK entrypoint: `infrastructure/bin/app.ts`
- `infrastructure/docker/Dockerfile` — PHP 8.3 + Nginx multi-service image
- `infrastructure/docker/Dockerfile.placeholder` — serves 503 during cold-start / image absence
- `infrastructure/docker/entrypoint.sh` — container startup script
- `infrastructure/docker/nginx/web.conf` — Nginx site configuration
- `infrastructure/docker/supervisord/web.conf` — supervisord config managing PHP-FPM + Nginx
- GitHub Actions: `.github/workflows/ci-aws-build.yml` — Docker build + ECR push workflow
- GitHub Actions: `.github/workflows/ci-aws-deploy.yml` — ECS force-new-deployment via cross-account role
- IAM policies: `.github/iam-policies/codebuild-assume-role-policy.json`, `deployment-permissions.json`, `trust-policy.json`
- `TcPortalStagingDeploymentRole` IAM role created with `ecs:UpdateService` + ECR verify permissions
- `infrastructure/cdk.context.json` — VPC/subnet/AZ context for CDK synthesis
- `infrastructure/package.json` — CDK deps: `aws-cdk-lib@^2`, `constructs@^10`
- `infrastructure/.gitignore` — excludes `cdk.out/` CDK build artifacts
