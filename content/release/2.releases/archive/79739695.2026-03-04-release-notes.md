## TL;DR

This release delivers the completed AWS cloud infrastructure and automated deployment pipeline for TC Portal. The application is now packaged as a Docker image and deployed to Amazon ECS Fargate — a fully managed container platform — with separate services for the web application, background job workers, the task scheduler, and Typesense search. Every push to the staging branch automatically triggers a Docker build and deployment via GitHub Actions, eliminating manual deployment steps and providing a repeatable, auditable path to production.

---

## What's New?

::tabs{default="Business View"}
:::tab{label="Business View"}

### Cloud Hosting Infrastructure Deployed to AWS

TC Portal now has a fully automated, code-defined cloud hosting environment on Amazon Web Services, ready to support reliable and scalable deployments.

**Why this matters**

Infrastructure that exists only in someone's head — or as manual steps in a runbook — creates serious operational risk. It cannot be audited, replicated quickly, or confidently handed over. By defining all infrastructure as versioned code, every component is now documented, reviewable, and reproducible, dramatically reducing the chance of environment drift, configuration mistakes, or knowledge loss.

**What's changing**

The platform now has a complete AWS hosting stack defined using AWS CDK (infrastructure-as-code): a network layer (VPC), containerised application services (ECS Fargate), database (RDS MySQL), caching (ElastiCache Redis), file storage (S3 and EFS), and observability (CloudWatch monitoring and alarms). The application is packaged as a Docker image and deployed automatically via GitHub Actions whenever code is pushed to staging.

**Who's affected**

Engineering and DevOps teams are the primary beneficiaries. Business leadership should be aware this represents the cloud migration milestone — the infrastructure is now production-ready for staging. The portal experience for end users is unchanged.

**Action required**

No action required from business leadership at this time. The infrastructure targets the staging sandbox environment. Production migration will be planned and communicated separately.

### Infrastructure Release — No User-Facing Changes

This release contains only infrastructure and CI/CD pipeline code. No features, workflows, or screens in TC Portal have changed.

**Why this was changed**

The engineering team has established cloud-native hosting infrastructure on AWS. This foundational work enables automated deployments and lays the groundwork for improved reliability and scalability.

**What's different**

Nothing visible to users has changed. The portal interface, navigation, data, and all existing functionality remain identical.

**Where in the platform**

Not applicable.

**How to help users**

If users report unexpected behaviour following this release, it is very unlikely to be related to these changes. Escalate through normal support channels.

**Likely Q&A**

> **Q: "Something looks different — was there an update?"**
> A: This release contained only infrastructure changes. Nothing in the portal interface was modified. If something appears different, it may be a browser cache issue — ask them to do a hard refresh (Ctrl+Shift+R or Cmd+Shift+R) and try again.

> **Q: "Is there going to be any downtime for this release?"**
> A: No planned downtime is expected. ECS Fargate deployments use a rolling update strategy — new containers start before old ones are stopped, so there should be no interruption for users.

> **Q: "Will my data be affected?"**
> A: No. This is purely an infrastructure change. No data migrations, schema changes, or user-facing modifications are included.

> **Q: "What is ECS Fargate / AWS CDK?"**
> A: These are cloud hosting technologies — think of them as the "engine room" that runs TC Portal on Amazon's servers. Users don't interact with these directly; they're simply how the application is hosted and deployed.

  ### Change Log

**[#6238](https://github.com/Trilogy-Care/tc-portal/pull/6238) — feat: AWS CDK infrastructure for ECS Fargate deployment**

- New CDK two-stack pattern: `FoundationStack` and `ApplicationStack` (TypeScript, AWS CDK)
- **Foundation Stack** (`infrastructure/lib/foundation-stack.ts`): VPC (2 AZs, public + private subnets), RDS MySQL 8.0, ElastiCache Redis 7.x, S3 bucket
- **Application Stack** (`infrastructure/lib/application-stack.ts`): ECS cluster, Application Load Balancer, 4 Fargate services (Web, Worker/Horizon, Scheduler, Typesense), auto-scaling, CloudWatch alarms + dashboard
- New Dockerfile (`infrastructure/docker/Dockerfile`): multi-stage PHP-FPM + Nginx image, Composer install, npm build, Laravel bootstrap
- Placeholder Dockerfile (`infrastructure/docker/Dockerfile.placeholder`): lightweight health-check image for initial ECS service registration
- Entrypoint script (`infrastructure/docker/entrypoint.sh`): runs migrations, seeds, cache warmup, then starts supervisord
- Nginx config (`infrastructure/docker/nginx/web.conf`): serving Laravel app on port 80 via PHP-FPM socket
- Supervisord config (`infrastructure/docker/supervisord/web.conf`): manages `php-fpm` and `nginx` processes in one container
- New GitHub Actions workflow: `ci-aws-build.yml` — triggers on `staging` branch pushes for app code paths; auto-increments semver tag (`v*.*.*`), builds Docker image, pushes to ECR (`trilogycare/tc-portal-stage`); runs on CodeBuild runner
- New GitHub Actions workflow: `ci-aws-deploy.yml` — triggers after successful build; deploys to ECS via `force-new-deployment`; uses cross-account `STS AssumeRole` (`TcPortalStagingDeploymentRole` on sandbox account `935598634652`)
- IAM policy: `.github/iam-policies/codebuild-assume-role-policy.json` — allows CodeBuild in DevOps account (`967883357946`) to assume `TcPortalStagingDeploymentRole`
- IAM policy: `.github/iam-policies/deployment-permissions.json` — ECS, ECR read, CloudWatch Logs, `iam:PassRole` permissions for ECS deployment
- IAM trust policy: `.github/iam-policies/trust-policy.json` — trusts DevOps account root + CodeBuild service with `aws:SourceAccount` condition
- CDK config: `infrastructure/cdk.json` — app entry `bin/app.ts`, feature flags for CDK v2 compatibility
- CDK context: `infrastructure/cdk.context.json` — availability zone lookup cache for `ap-southeast-2`
- CDK app entry: `infrastructure/bin/app.ts` — instantiates `FoundationStack` then `ApplicationStack` with cross-stack references
- EFS persistence: access point + IAM auth for Typesense data directory
- Cross-account ECR image pull from DevOps account (`967883357946`) to sandbox (`935598634652`)
- CloudWatch: Container Insights enabled, per-service CPU/memory alarms, operations dashboard
- Target: sandbox account `935598634652` (staging only; production migration separate)
- `infrastructure/package.json`: CDK v2 dependencies (`aws-cdk-lib`, `constructs`)
- `infrastructure/tsconfig.json`: strict TypeScript config targeting ES2020
- `infrastructure/.gitignore`: excludes `cdk.out/` build artifacts
