## TL;DR

This release contains a targeted but important overhaul of the MYOB sub-account transaction synchronisation process. The previous implementation deleted transactions too aggressively — records already linked to funding entries could be removed and lost. The rewritten job now preserves those records and adds support for Voluntary Contribution funding updates and bill payment tracking during sync. Two deprecated MYOB API field fallbacks have been removed, and a GST account code has been split into two codes to correctly distinguish Home Care Package and Support at Home programmes.

### Safer MYOB Transaction Synchronisation

The process that pulls financial transactions from MYOB into the portal has been overhauled to prevent data loss during sync operations.

**Why this matters**

The previous sync job deleted transactions broadly before re-importing them from MYOB. This could inadvertently remove transaction records that were already linked to participant funding entries, creating gaps in financial data and requiring manual recovery. The risk was highest for organisations running the sync frequently or for large sub-account sets.

**What's changing**

The sync job now only removes transactions that have not yet been linked to funding records, preserving all existing financial linkages. It also correctly handles Voluntary Contribution transactions during sync — ensuring VC funding balances reflect MYOB data accurately. When MYOB indicates a bill has been paid, the portal bill status is updated accordingly during the sync cycle.

**Who's affected**

Finance teams relying on MYOB transaction data in the portal for reconciliation and funding balance accuracy. No action is required from users.

**Action required**

No action required. This improvement applies automatically to all future MYOB sync operations.

---

### GST Account Code Split: HCP and SAH

The portal now distinguishes between two GST account codes — one for Home Care Package billing and one for Support at Home — improving financial reporting accuracy.

**Why this matters**

Previously, a single GST account code (`1500`) was used for both Home Care Package and Support at Home programmes. As Support at Home becomes the primary funding model, this conflation created reconciliation ambiguity. Separating the codes ensures financial reports can correctly attribute GST transactions to the right programme.

**What's changing**

The GST account code `1500` is now labelled as HCP-specific (`GST_PAID_HCP`), and a new code `1510` (`GST_PAID_SAH`) has been added for Support at Home. This affects internal financial reporting and GL code mapping — no user workflow changes.

**Who's affected**

Finance teams reviewing GST transaction breakdowns in the portal.

**Action required**

No action required. The change takes effect automatically.

### MYOB Transaction Sync Reliability Fix

The MYOB sub-account transaction sync job has been rewritten to prevent data loss and improve accuracy.

**Why this was changed**

The previous sync job deleted all matching transactions before re-importing them from MYOB. This was problematic if any of those transactions were already linked to funding entries in the portal — those links would be lost. The fix ensures only unlinked transactions are removed during a sync cycle.

**What's different**

Before: all transactions for a sub-account were deleted and re-created from MYOB on every sync. After: only transactions not yet linked to funding entries are deleted; linked ones are preserved. Additionally, VC funding balances and bill payment statuses are now updated during sync.

**Where in the platform**

Background job — not visible in the portal UI.

**How to help users**

If a user reports that MYOB transaction data appears inconsistent or that previously visible transactions have disappeared: note that this has been addressed in this release. If the issue persists after the fix, escalate with the specific sub-account and transaction details.

**Likely Q&A**

> **Q: "Some MYOB transactions are missing from the portal — they were there before?"**
> A: This was a known risk in the previous sync implementation where transactions could be removed if a re-sync ran while they were linked to funding entries. The fix in this release prevents that from happening going forward. For historically missing data, the MYOB sync can be re-run for the affected sub-account to restore records.

> **Q: "A bill that was paid in MYOB still shows as unpaid in the portal?"**
> A: The sync job now updates bill payment status during the sync cycle. If the bill was paid in MYOB after the last sync ran, it will be updated on the next sync. If it is still not updating, escalate with the bill ID and MYOB reference number.
