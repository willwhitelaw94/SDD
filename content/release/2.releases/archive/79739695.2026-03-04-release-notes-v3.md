## TL;DR

This release delivers a broad set of billing, supplier, and compliance improvements. Care coordinators and suppliers can now see on-hold and rejection reasons directly on bills. The Bills table gains fee exemption filters and columns to help coordinators identify exempt clients quickly. A team permissions hotfix corrects access control for team-based workflows. The Budgets Reloaded project lays its backend and frontend foundation. Organisation short codes and brand colours are generated automatically to power upcoming UI redesigns. Several performance optimisations reduce database query counts on hot paths, and compliance staff gain a new permission to add missing supplier documents.

## What's New?

:::tab{label="Business View"}

### Billing Transparency: On-Hold and Rejection Reasons Now Visible

Care coordinators and suppliers can now see the specific reason a bill has been placed on hold or rejected, directly on the bill detail screen.

**Why this matters**

Previously, when a bill was placed on hold or rejected by the case manager, the reason was only visible internally. Coordinators and suppliers had to contact the case management team to understand what action was required. This created unnecessary back-and-forth and delayed resolution.

**What's changing**

The on-hold or rejection reason is now displayed prominently on the bill for all authorised viewers — care coordinators and suppliers alike. The internal comment field has also been extended to accommodate longer explanations where needed.

**Who's affected**

Care coordinators, suppliers, and case managers reviewing bills.

**Action required**

No action required. The change is applied automatically to all existing and future bills.

---

### Bills Table: Fee Exemption Filters and Columns

The Bills table now includes columns and filters for loading fee exemptions, making it easy to identify clients whose bills are exempt from coordination overhead fees.

**Why this matters**

Coordinators managing clients with fee exemptions needed to manually cross-reference information to identify affected bills. This adds native filtering and visibility directly in the table.

**What's changing**

Two new columns are available in the Bills table: exemption status and the staff member who applied the exemption. Corresponding filter controls allow coordinators to narrow the table to exempt bills only.

**Who's affected**

Care coordinators and billing staff.

---

### Compliance: New Permission to Add Missing Supplier Documents

Compliance managers and team members can now add missing documents to supplier profiles through the portal, without requiring administrator intervention.

**Why this matters**

Compliance teams were blocked from uploading missing supplier documents when a supplier had not yet self-served the upload. This required escalation to administrators, slowing down verification workflows.

**What's changing**

A new `can-add-supplier-documents` permission has been added and assigned to the Admin, Compliance Manager, and Compliance Team Member roles.

**Who's affected**

Compliance managers and compliance team members.

---

### Team Permissions Hotfix

A permissions issue affecting team-based workflows has been resolved. Users with team-scoped access were encountering incorrect permission denials in certain scenarios.

**Why this matters**

Incorrect permissions prevent authorised users from performing their work and erode trust in the platform.

**What's changing**

The permission evaluation logic for team contexts has been corrected. Affected users will now see the correct access behaviour without needing any changes on their end.

**Who's affected**

Any users working within team-based access structures.

---

### Organisation Short Codes and Brand Colours

Verified organisations now have automatically generated short codes (3–4 character identifiers) and deterministic brand colours assigned. These identifiers will power upcoming visual redesigns, including the Budgets Reloaded project.

**Why this matters**

The upcoming Budgets Reloaded UI redesign requires visual identifiers for organisations to provide clear, colour-coded context throughout the budget screens.

**What's changing**

Each verified organisation receives a unique short code and a consistent brand colour. These are generated automatically and backfilled for existing organisations. No manual configuration is required.

**Who's affected**

All verified organisations. The identifiers are not yet visible in the current UI — they will surface in upcoming feature releases.

---

### Supplier Dashboard: Expired Document Alert Fix

The expired document alert on the supplier dashboard now correctly shows only when a supplier has genuinely expired documents, rather than appearing incorrectly in some edge cases.

**Why this matters**

False alerts on the supplier dashboard caused confusion and unnecessary support contacts from suppliers who had no actual expired documents.

**What's changing**

The alert logic has been corrected. Suppliers with no expired documents will no longer see the alert.

**Who's affected**

Suppliers.

---

### Supplier Verification: Police Check and NDIS Check Logic Corrected

The logic determining whether a supplier is verified now correctly accounts for police checks and NDIS worker screening checks as valid verification methods.

**Why this matters**

Some suppliers with valid NDIS or police checks were being incorrectly flagged as unverified, creating friction in the onboarding workflow.

**What's changing**

The verification check has been updated to include `POLICE_CHECK` or NDIS screening as valid verification paths. Affected suppliers will automatically reflect the correct verified status.

**Who's affected**

Suppliers and compliance teams.

---

### Supplier Invitation: Portal Stage Always Set to Invited

When a staff member invites a supplier, the portal stage is now always correctly set to `INVITED`, regardless of the supplier's prior state.

**Why this matters**

In some cases, re-inviting a supplier did not correctly reset the portal stage, leading to inconsistent state and potential workflow issues.

**What's changing**

The `inviteSupplier` method now explicitly forces the portal stage to `INVITED` on every invitation, ensuring consistent behaviour.

**Who's affected**

Staff members inviting suppliers.

---

### Budgets Reloaded: Foundation Work

The Budgets Reloaded project has begun. This release establishes the backend data structures, API endpoints, and frontend scaffolding that will support a redesigned budgets experience.

**Why this matters**

The current budget screens are being redesigned to improve usability and incorporate new funding stream data. This release lays the foundations — no new screens are visible to users yet.

**What's changing**

Backend: new data classes, actions, and API routes for funding streams and budget plan groupings. Frontend: structural scaffolding for the upcoming redesigned budget pages. A new API endpoint (`GET /budget/{id}/funding-streams`) is available for internal integrations.

**Who's affected**

No end users are affected by this release. The changes are internal and structural.

#### Enhancements

**[#6186](https://github.com/Trilogy-Care/tc-portal/pull/6186) — DUC-179: DB hot-path query optimization**
`Oscar-BK`

Five separate optimisations targeting the most expensive queries per request:

1. **Notification queries** — `simplePaginate()` switched to `cursorPaginate()`; unread count computed only on first page (no `COUNT(*)` on subsequent pages); `markAllRead` changed from model hydration to single `UPDATE` statement.
2. **HandleInertiaRequests** — All shared props (profile, sidebar, permissions, intercom) wrapped in `fn () =>` closures; non-Inertia XHR responses skip evaluation entirely. Saves 12–18 DB queries per non-Inertia request.
3. **nova_get_setting cache** — `config/nova-settings.php` now includes `cache` key with `:memory:` driver (per-request singleton). Eliminates repeated DB hits for the same setting key within a request.
4. **CheckPermission memoisation** — `isEligibleCareCoordinator()` memoised with `private static array $eligibleCache` keyed by `$userId`. Deduplicates the raw DB query within a single request.
5. **Missing database indexes** — Two new composite covering indexes:
   - `organisation_workers(user_id, workerable_type, workerable_id)` — for eligibility check
   - `user_verification_logs(user_id, type, status, remember, created_at)` — for verification lookups

Migrations: `2026_02_24_104239_add_composite_index_to_notifications_table`, `2026_02_24_133511_add_composite_indexes_to_organisation_workers_and_verification_logs_table`

10 new Pest tests across `NotificationControllerTest`, `IsEligibleCareCoordinatorMemoizationTest`, `OptimizationIndexesMigrationTest`.
