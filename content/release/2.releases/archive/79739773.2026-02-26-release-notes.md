## TL;DR

This release ships a completely redesigned statement generation engine that replaces six scattered background jobs with a unified, trackable StatementRun system — giving operations teams full visibility into statement progress and the ability to target any billing month. Alongside this, critical fixes to the team permission system resolve duplicate key violations and ensure role-based access works correctly whether or not a user is scoped to a team. The admin dashboard now loads reliably after database indexes were added to eliminate a MySQL temp table overflow error.

## What's New?

::tabs{default="Business View"}
:::tab{label="Business View"}

### Statement Generation Now Fully Trackable

Statement production has been overhauled with a new run-based engine that gives operations complete visibility into every step of the process.

**Why this matters**

Previously, statement generation ran as a collection of opaque background jobs with no centralised status tracking. If a run failed partway through, it was difficult to know which packages were affected or how to recover. This creates operational risk at end-of-month when statements are time-sensitive.

**What's changing**

Statement generation is now orchestrated through a StatementRun — a single tracked entity that records total packages to process, how many have been completed, status (pending → processing → completed/failed), and who initiated the run. A new Finance → Statements screen provides a live view of all runs. The system also supports targeted regeneration for specific clients and batches statements for physical mail delivery through SmartComm.

**Who's affected**

Finance and operations staff responsible for end-of-month statement generation. No changes to what recipients or coordinators receive — only how the process is managed internally.

**Action required**

Statement generation should now be initiated through the new Finance → Statements interface. Brief the operations team on the new screen before the next statement cycle.

---

### Team Permission System Stabilised

A critical fix to the permission infrastructure prevents access control failures for users who belong to teams.

**Why this matters**

Since team scoping was introduced, users in team contexts were intermittently losing access to actions they should have had — because the permission check was not correctly falling back to globally scoped roles. In some cases, assigning a role caused a duplicate key database violation. These are correctness and reliability issues that directly affect staff ability to do their jobs.

**What's changing**

The permission resolution logic has been overhauled so that global role permissions (assigned at team_id = 0) are always checked regardless of the team context a user is currently in. Duplicate key violations on role assignment are prevented by fixing the database primary keys to include team_id. The system now initialises with a safe default global context at startup and in queued jobs.

**Who's affected**

Any staff user who belongs to a team. Administrators who assign roles.

**Action required**

No action required — the fix is automatic. Monitor for any unexpected permission behaviour in the first week and escalate if found.

---

### Admin Dashboard Performance Restored

The admin dashboard was failing to load for some organisations due to a database query consuming excessive temporary storage.

**Why this matters**

An unresponsive admin dashboard prevents supervisors from monitoring care coordinator activity and package stages, creating a blind spot in operational oversight.

**What's changing**

Two database indexes have been added to the stage histories table, allowing MySQL to answer the affected query efficiently without creating large temporary tables.

**Who's affected**

Administrators using the admin dashboard, particularly for organisations with high stage history volumes.

**Action required**

No action required.

### New Finance → Statements Page

A dedicated Statements section is now available in the Finance menu for staff with access.

**Why we built this**

The previous statement generation process had no interface for tracking progress — staff had to rely on background job logs to know if generation was complete. This made the end-of-month process stressful and opaque.

**What's changed**

There is now a full Statements management page where you can see all statement runs, their status, how many packages have been processed, and when they completed.

**Where to find it**

`Finance → Statements`

**How to use it**

1. Go to `Finance → Statements` in the left sidebar
2. View existing statement runs and their current status
3. Initiate a new statement run as required for the billing month

If the Statements menu item is not visible, you may not have the required access level — speak to your administrator.

### New Finance → Statements Interface

A new statements management interface is available to staff with Finance access.

**Why this was changed**

Statement generation previously had no visible status tracking. The new interface gives operations a central place to manage and monitor all statement runs.

**What's different**

Previously there was no dedicated Statements page. Now `Finance → Statements` shows all statement runs with real-time status (pending, processing, completed, failed) and package counts.

**Where in the platform**

`Finance → Statements` — visible in the left sidebar under Finance for authorised staff.

**How to help users**

If staff cannot see the Statements menu item, check their role and permissions. The page requires Finance-level access.

**Likely Q&A**

> **Q: "I can't see the Statements menu item under Finance?"**
> A: The user may not have the required permissions. Check their assigned role — Finance access is needed. If they should have access, ask an admin to verify their permissions.

> **Q: "A statement run shows as 'failed' — what do we do?"**
> A: Capture the statement month and run ID and escalate to the development team. The run can be re-initiated once the cause is identified.

> **Q: "Why is there a new statement run every time we generate?"**
> A: Each generation creates a new run record for auditability. Previous runs remain visible for historical reference.

### Team Permissions Fix

Staff who belong to teams should now have consistent access to their expected features.

**Why this was changed**

A bug in the team permission system was causing global role permissions to be ignored when a user had a team context set. This could result in staff being unexpectedly denied access to actions their role normally permits.

**What's different**

The permission resolution now correctly checks both team-scoped and global permissions. Staff should experience no further unexpected access denials related to team membership.

**Where in the platform**

Platform-wide — affects any permission check for users in teams.

**How to help users**

If a user reports unexpected access denials, first check whether their role has changed. If their role is correct and they are a member of a team, the fix in this release should have resolved it. If issues persist after this release, escalate with the user's ID, role, and the specific feature they cannot access.

**Likely Q&A**

> **Q: "I'm in a team but I can't access something I could before?"**
> A: This release includes a fix for exactly this issue. If the problem persists after the deployment, capture the specific page or action and escalate to support with the user's role and team details.

> **Q: "An admin got a database error when assigning a role?"**
> A: A migration in this release fixes the underlying primary key issue. This should no longer occur after deployment.

---

### Admin Dashboard Now Loading

The admin dashboard should now load correctly for all organisations.

**Why this was changed**

A query on the stage histories table was consuming so much temporary database space that MySQL ran out of room, causing the admin dashboard to fail.

**What's different**

Database indexes have been added so the query runs efficiently. The dashboard should now load without errors.

**Where in the platform**

Admin dashboard.

**Likely Q&A**

> **Q: "The admin dashboard is still not loading?"**
> A: Confirm the deployment has been completed. If the error persists post-deployment, capture the exact error message and escalate with the organisation ID.
