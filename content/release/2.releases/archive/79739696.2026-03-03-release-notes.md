## TL;DR

This release establishes TC Portal's cloud-native hosting foundation on AWS, running the application as containerised services on Amazon ECS Fargate. The entire infrastructure is defined as code using AWS CDK, enabling repeatable and auditable deployments. A fully automated CI/CD pipeline via GitHub Actions now builds Docker images and deploys the staging environment whenever application code changes are pushed, removing manual deployment steps and reducing the risk of human error during releases.

---

## What's New?

::tabs{default="Business View"}
:::tab{label="Business View"}

### Cloud-Native Infrastructure Established on AWS

TC Portal now has a fully defined, automated cloud hosting environment on Amazon Web Services that can be deployed, replicated, and managed with confidence.

**Why this matters**

Maintaining infrastructure through manual processes or ad-hoc configuration creates operational risk — changes are hard to audit, difficult to reproduce, and prone to inconsistency across environments. By encoding all infrastructure as versioned code, this work eliminates that risk. Every server, network, database, and deployment pipeline is now documented, reviewable, and repeatable.

**What's changing**

The platform's hosting infrastructure has been defined using AWS CDK (infrastructure-as-code), specifying the network, compute, database, caching, and storage layers in a structured format. A complete Docker packaging of the application and an automated build-and-deploy pipeline have also been established. This is the foundation on which reliable, scalable cloud deployments will run.

**Who's affected**

Engineering and DevOps teams are the primary stakeholders. Leadership should be aware this represents a significant milestone in the cloud migration roadmap. No changes are visible to end users of the portal at this stage.

**Action required**

No action required from business leadership at this time. The infrastructure targets the staging environment (AWS sandbox account). Production deployment will follow as a separate, planned migration step.

  ### Change Log

**[#6238](https://github.com/Trilogy-Care/tc-portal/pull/6238) — feat: AWS CDK infrastructure for ECS Fargate deployment**

New AWS CDK project (`infrastructure/`) written in TypeScript, implementing a two-stack pattern for TC Portal's cloud-native hosting:

- New CDK entry point: `infrastructure/bin/app.ts` — instantiates `FoundationStack` and `ApplicationStack` in sandbox account `935598634652` (ap-southeast-2)
- New `FoundationStack` (`infrastructure/lib/foundation-stack.ts`):
  - VPC: `tc-portal-vpc`, 2 AZs, public/private/isolated subnets, configurable NAT gateways
  - Security groups: `tc-portal-alb-sg`, `tc-portal-ecs-sg`, `tc-portal-database-sg`, `tc-portal-redis-sg`
  - RDS MySQL 8.0 (`db.t3.micro` default, multi-AZ optional), credentials stored in Secrets Manager
  - ElastiCache Redis 7.x (`cache.t3.micro` default)
  - S3 bucket for file uploads (versioning enabled)
- New `ApplicationStack` (`infrastructure/lib/application-stack.ts`):
  - ECS cluster `tc-portal-{env}-cluster` with Container Insights V2 enabled
  - Private DNS namespace `tc-portal-{env}.local` (AWS Cloud Map service discovery)
  - ALB `tc-portal-{env}-alb`: HTTP listener with optional HTTPS/cert redirect
  - 4 Fargate services:
    1. **Web** — Nginx + PHP-FPM via Supervisord; exposed on port 80; auto-scales on CPU & memory
    2. **Worker** — Laravel Horizon (`php artisan horizon`); queue processing
    3. **Scheduler** — Laravel `php artisan schedule:work`; cron replacement
    4. **Typesense** — self-hosted search engine on port 8108; EFS-backed data persistence
  - EFS `TypesenseFileSystem`: encrypted, `RETAIN` removal policy, access point at `/typesense` (UID/GID 1000)
  - Cross-account ECR: images pulled from DevOps account `967883357946` (`trilogycare/tc-portal-stage`)
  - Secrets Manager references (not created by CDK): `tc-portal/app-key`, `tc-portal/sentry`, `tc-portal/sidecar`, `tc-portal/twilio`, `tc-portal/openai`, `tc-portal/klaviyo`, `tc-portal/posthog`
  - CloudWatch alarms: CPU >70%, memory >80%, ALB 5xx errors >5/min, unhealthy target hosts; SNS email notifications
  - CloudWatch dashboard: CPU, memory, request count, latency, 5xx errors per service
- New Docker packaging (`infrastructure/docker/`):
  - `Dockerfile`: multi-stage build — Composer deps (PHP 8.3-fpm-alpine base) → Node.js build → final Nginx+PHP-FPM image
  - `Dockerfile.placeholder`: lightweight placeholder image for initial stack deployment
  - `entrypoint.sh`: runs `php artisan migrate --force`, `config:cache`, `route:cache`, `view:cache` on startup
  - `nginx/web.conf`: Nginx server block, `try_files` routing, PHP-FPM fastcgi pass to `127.0.0.1:9000`
  - `supervisord/web.conf`: supervises `nginx` and `php-fpm` processes
  - `.dockerignore`: excludes `node_modules`, `.git`, `infrastructure/`, tests, IDE config
- New GitHub Actions build workflow (`.github/workflows/ci-aws-build.yml`):
  - Triggers on push to `staging` for paths: `app/**`, `domain/**`, `config/**`, `routes/**`, `resources/**`, `database/**`, `public/**`, `app-modules/**`, `infrastructure/docker/**`, `composer.*`, `package*.json`, `vite.config.js`
  - Runs on CodeBuild runners (`codebuild-trilogy-care-organisation-build-*`)
  - Auto-increments git tag (e.g. `v2.0.51` → `v2.0.52`), pushes tag to origin
  - Builds Docker image, tags with short git SHA and `latest`
  - Pushes to ECR repo `trilogycare/tc-portal-stage` in DevOps account `967883357946`
  - Restricted to `staging` branch only
- New GitHub Actions deploy workflow (`.github/workflows/ci-aws-deploy.yml`):
  - Triggers automatically after successful build workflow, or manually via `workflow_dispatch`
  - Assumes cross-account IAM role `TcPortalStagingDeploymentRole` in sandbox account `935598634652` via OIDC
  - Verifies ECR image exists before deploying
  - Forces new ECS deployments for all 4 services: Web, Worker, Scheduler, Typesense
  - Waits for service stability with 15-minute timeout
  - Posts deployment summary to GitHub step summary
- New IAM policies (`.github/iam-policies/`):
  - `codebuild-assume-role-policy.json`: `sts:AssumeRole` on `arn:aws:iam::935598634652:role/TcPortalStagingDeploymentRole`
  - `deployment-permissions.json`: ECS (`UpdateService`, `DescribeServices`, `RegisterTaskDefinition`, `DeregisterTaskDefinition`), ECR read, CloudWatch Logs write — scoped to `tc-portal-*` resources in `ap-southeast-2`
  - `trust-policy.json`: trust relationship allowing CodeBuild from DevOps account to assume the deployment role
- CDK project config: `infrastructure/package.json` (aws-cdk-lib 2.x, constructs 10.x), `infrastructure/tsconfig.json`, `infrastructure/cdk.json`, `infrastructure/cdk.context.json`, `infrastructure/.gitignore`
