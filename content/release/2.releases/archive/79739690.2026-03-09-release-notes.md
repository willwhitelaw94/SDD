## TL;DR

- **Team permissions restored** — A critical fix ensures user role permissions are correctly enforced across team and global contexts, resolving incorrect access denials and permission assignment errors across the portal.
- **Organisation short codes & brand colours** — Each organisation can now be assigned a 4-character short code and a brand colour, improving identification in reports and throughout the portal interface.
- **Bill on-hold reasons now visible** — Care Coordinators and Suppliers can now see the reason a bill was placed on hold or rejected directly on the bill, eliminating the need to contact internal teams for clarification.
- **Supplier verification logic improved** — Suppliers are now correctly verified if they hold either a Police Check or an NDIS Worker Screening check, fixing a rule that previously required only Police Check.
- **Compliance team can add missing supplier documents** — A new permission allows Compliance Managers and Compliance Team Members to upload missing supplier documents directly, without escalation to Admin.
- **Significant performance improvements** — Multiple database optimisations including composite indexes, deferred shared props, and query memoisation reduce page load times and server overhead across the portal.

### Team Permissions Fix

A critical issue with how team-scoped permissions were evaluated has been resolved, restoring correct role-based access control across the platform.

**Why this matters**

Incorrect permission enforcement is a compliance and governance risk. Users were experiencing access denials or unexpected permission grants depending on whether a team context was active. This fix restores the intended behaviour where global role permissions apply correctly regardless of team context, and eliminates duplicate key database errors when assigning roles.

**What's changing**

The platform now correctly merges global (cross-team) and team-scoped permissions, sets a safe default team context for background jobs and CLI commands, and prevents constraint violations when roles are assigned. There is no change to the intended permission model — only the enforcement logic has been corrected.

**Who's affected**

Any user working within team-structured organisations. Admin users responsible for role and permission management.

**Action required**

No action required — the fix is applied automatically. Admins should verify that user access looks as expected after this release. If any users report unexpected permission changes, contact the support team.

### Organisation Short Codes & Brand Colours

Organisations can now be assigned a unique 4-character short code and a hex brand colour, providing a compact, visually distinct identifier for use in reports and portal interfaces.

**Why this matters**

As the number of organisations on the platform grows, having a standardised short-form identifier improves data readability in reports and facilitates faster identification when staff are working across multiple organisations.

**What's changing**

Each organisation record now supports a `code` (4 characters, auto-generated) and a `colour` (hex value, auto-generated). These fields are designed to be consistent and unique. The generation logic ensures codes and colours are assigned deterministically.

**Who's affected**

Internal staff and administrators who work with multiple organisations. The fields will surface in portal views and reporting where organisation identity is displayed.

**Action required**

No action required — codes and colours will be assigned automatically for existing and new organisations. Admin users can review and update them if needed.

### Bill On-Hold Reasons Now Visible

When a bill is placed on hold or rejected, the reason is now displayed directly on the bill for Care Coordinators and Suppliers.

**Why this matters**

Previously, when a bill was held or rejected, the responsible coordinator or supplier had no visibility into why — requiring back-and-forth communication with internal teams. This change eliminates that overhead and accelerates bill resolution.

**What's changing**

The on-hold or rejection reason now appears inline on the bill view for Care Coordinators and Suppliers. Additionally, when a Care Management bill is placed on hold due to a validation error, the specific validation error detail is now included in the comment rather than a generic "see logs" message.

**Who's affected**

Care Coordinators, Suppliers, and any staff involved in bill review and approval workflows.

**Action required**

No action required — the reason field is now automatically visible. Staff should ensure on-hold reasons are written clearly, as they are now visible externally.

### Supplier Verification Logic Improved

The portal now correctly verifies JUST_ME suppliers by accepting either a Police Check or an NDIS Worker Screening check — whichever is present and verified.

**Why this matters**

The previous logic required a Police Check specifically, which incorrectly blocked suppliers who had completed an NDIS Worker Screening check instead. This was causing legitimate suppliers to be held at an incorrect portal stage, creating compliance friction without regulatory basis.

**What's changing**

The verification check now applies OR logic — a JUST_ME supplier is verified if they have either a Police Check or an NDIS Worker check that is present and verified. All existing required documents (Public Liability Insurance) are unaffected.

**Who's affected**

JUST_ME suppliers who previously could not progress through the portal despite holding an NDIS Worker Screening check. Compliance team members who manage supplier onboarding.

**Action required**

Compliance and onboarding teams should review any JUST_ME suppliers currently held at an incorrect stage. This fix will automatically unblock those with valid NDIS worker checks.

### Compliance Team Can Add Missing Supplier Documents

A new portal permission allows Compliance Managers and Compliance Team Members to upload missing supplier documents directly, without requiring Admin escalation.

**Why this matters**

Previously, only Admins could add supplier documents. Compliance staff had to request document uploads through an Admin, creating unnecessary bottlenecks in compliance reviews and supplier onboarding.

**What's changing**

A new `can-add-supplier-documents` permission is now available. It is assigned to: Admin, Compliance Manager, and Compliance Team Member roles.

**Who's affected**

Compliance Managers and Compliance Team Members who review supplier documentation.

**Action required**

No action required — the permission is automatically assigned to the relevant roles.

### Performance Improvements Across the Portal

A comprehensive set of database and application optimisations has been implemented, reducing page load times and server resource usage across the portal.

**Why this matters**

Slower page loads directly affect staff productivity and create a poor experience for external users (suppliers, coordinators). These optimisations address several identified hot-path bottlenecks that were generating unnecessary database queries on every request.

**What's changing**

Key improvements include: composite database indexes on frequently queried tables, deferred evaluation of shared page props (reducing unnecessary queries on non-page requests), per-request caching of repeated settings lookups, and memoisation of eligibility checks called multiple times per request.

**Who's affected**

All portal users — improvements are transparent and require no behaviour change.

**Action required**

No action required — performance improvements are applied automatically.

### Team Permissions Fix

A critical fix has been applied to how role-based permissions work when team contexts are involved.

**Why this was changed**

Users in team-structured organisations were experiencing incorrect access denials or unexpected permission grants. The issue was in how Spatie's permission system resolved global vs team-scoped permissions.

**What's different**

- Global (cross-team) permissions now correctly apply even when a team context is active
- Role assignment no longer triggers duplicate key database errors
- Background jobs and CLI commands now use a safe default team context (0 = global)
- The `CheckPermission` middleware and User model permission methods have been updated

**Where in the platform**

Affects all permission-gated areas of the portal. Most visible in team-structured organisations where users have team-specific roles.

**How to help users**

If a user reports they can't access something they previously could (or vice versa), check their role assignments via Admin → Users. If permissions look correct but behaviour is wrong, escalate — this may indicate a residual issue.

**Likely Q&A**

> **Q: "I can't access [feature] anymore after this release?"**
> A: Check the user's role and team assignments. The fix may have corrected a previously incorrect permission grant. If access should be restored, an Admin can review and update the role.

> **Q: "A user is getting a 'permission denied' error they weren't seeing before?"**
> A: This is likely the permissions now correctly enforcing what was always intended. Verify the role grants the required access and escalate if the role seems correct but access is still denied.

### Supplier Verification Logic Change (POLICE_CHECK or NDIS)

The logic for verifying JUST_ME suppliers has been updated to accept either a Police Check or an NDIS Worker Screening check.

**Why this was changed**

Suppliers with a valid NDIS Worker Screening check were being blocked at the verification stage because the system only checked for a Police Check. This was incorrect — both documents serve the same verification purpose.

**What's different**

- Old behaviour: JUST_ME suppliers required a Police Check to pass verification.
- New behaviour: JUST_ME suppliers pass verification if they have either a Police Check OR an NDIS Worker Screening check (at least one must be present and verified).
- Public Liability Insurance is still always required.

**Where in the platform**

Supplier portal stage / compliance document review.

**How to help users**

If a JUST_ME supplier asks why they were previously blocked despite having an NDIS check, explain that this has now been fixed and they should be able to proceed.

**Likely Q&A**

> **Q: "I have an NDIS Worker check but the portal says I'm not verified?"**
> A: This should now be resolved. Ask them to log in and check their portal stage. If still blocked, verify the NDIS document is marked as verified (not just uploaded) in the system.

> **Q: "Does a Police Check still work?"**
> A: Yes — either a Police Check or an NDIS Worker check satisfies the requirement.
