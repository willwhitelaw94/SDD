## TL;DR

This release fixes a critical issue where users assigned to teams were being incorrectly denied access to features their role should have granted. A database schema migration is included that repairs the underlying permission table structure. Alongside this, the MYOB transaction sync system has been rebuilt to process one account at a time with proper timeout handling and preservation of funding-linked records. A small hotfix also ensures the BPay field is correctly passed when creating a supplier from within a bill.

## What's New?

::tabs{default="Business View"}
:::tab{label="Business View"}
### Team-Based Access Control Now Works Reliably

Staff assigned to operational teams can now access all features their role permits, resolving intermittent permission failures introduced with the team feature rollout.

**Why this matters**

The team feature was designed to allow staff to be grouped into operational units such as Bill Processing or care pods. A defect in the underlying permission infrastructure meant that when a team context was active, global role permissions were not correctly resolved — staff on teams were being incorrectly denied access to features. This fix is essential before teams can be safely adopted at scale.

**What's changing**

Permission checks now correctly consider both team-scoped and global role-based permissions. A database schema migration is included that reconfigures how permission assignments are stored, eliminating the risk of duplicate records or integrity violations when assigning roles to team members.

**Who's affected**

Any staff member assigned to a team. System administrators and operational leads responsible for access control configuration.

**Action required**

No action required — the fix applies automatically on deployment. If teams are being rolled out to additional staff, this fix must be in place first.

---

### MYOB Transaction Sync Rebuilt for Reliability

The background reconciliation process that syncs financial transactions from MYOB has been refactored to process each account individually, reducing the risk of timeouts and silent failures.

**Why this matters**

The previous batch processing approach could time out when syncing multiple MYOB sub-accounts together. Silent failures meant the Finance team could be working from incomplete transaction data without knowing. The refactored system processes each account independently, with explicit timeout controls and protection of funding-linked records during resync.

**What's changing**

Each MYOB sub-account is now synced via its own dedicated background job. Transactions linked to funding records are preserved during resync operations, and voluntary contribution entries are now correctly handled. The system is also more resilient to MYOB API slowness via configurable timeouts.

**Who's affected**

Finance team members and anyone monitoring MYOB reconciliation accuracy. The change is transparent to end users — no workflow changes are required.

**Action required**

If any automation or manual triggers reference the old command `sync:myob-transactions-by-sub-accounts` (plural), these should be updated to `sync:myob-transactions-by-sub-account` (singular, single account).

### Supplier BPay Flag Correctly Recorded on Bill Creation

When creating a supplier from within a bill, the BPay payment indicator is now correctly included in the submission.

**Why this matters**

If a supplier uses BPay as their payment method, this flag was being silently omitted when creating them via the bill creation flow, potentially resulting in incorrect payment routing records.

**What's changing**

The supplier creation form now defaults `is_bpay` to false and includes it in all submissions, ensuring the payment method field is always present.

**Who's affected**

Finance staff who create suppliers from within the bill creation workflow.

**Action required**

No action required — the fix applies automatically. Any suppliers created via this flow before this release should be reviewed to confirm payment details are accurate.

### Your Role Permissions Work Correctly When You're in a Team

If you've been assigned to a team in the portal, you should now be able to access all the features your role allows without running into unexpected "Access Denied" errors.

**Why we built this**

We found that users placed into teams were sometimes blocked from pages and actions they should have been able to access. The team feature is designed to group you with colleagues for operational purposes — it was never meant to restrict what you can do. We've fixed the underlying issue so your role permissions apply correctly whether you're in a team or not.

**What's changed**

No visible changes — things that should work now work. If you were experiencing unexpected permission errors after being assigned to a team, these should be resolved.

**Where to find it**

No specific location — this affects permission checks across the entire portal.

**How to use it**

No steps needed — this works automatically.

### BPay Correctly Set When Creating a Supplier from a Bill

When adding a new supplier directly within a bill, the BPay payment setting is now included correctly.

**Why we built this**

The BPay indicator was accidentally missing from the supplier creation form within the bill workflow, which could lead to incorrect payment details being saved for that supplier.

**What's changed**

The form now correctly defaults BPay to "off" and always includes it when submitting.

**Where to find it**

`Bills → Create Bill → Add Supplier`

**How to use it**

No steps needed — this works automatically when you create a supplier from a bill.

:::tab{label="Support View"}
### Team Permission Errors Resolved

Users assigned to teams were being incorrectly denied access to features their role should grant.

**Why this was changed**

The team feature groups staff into operational units. A defect in the permission system caused checks to fail when a team context was active, denying access to features permitted by the user's role.

**What's different**

Previously: Users on teams could encounter unexpected "Forbidden" or "Access Denied" errors even when their role should have allowed access.

Now: Permission checks correctly resolve both team-specific and global role permissions. Users on teams can access all features their role grants.

**Where in the platform**

Affects all permission-gated pages and actions across the portal — particularly areas restricted by role (Bills, Notes, Admin sections).

**How to help users**

If a user reports unexpected access errors after being assigned to a team, ask them to log out and back in to refresh their session. If issues persist after deployment, escalate to engineering with their user ID and the page or action they're trying to access.

**Likely Q&A**

> **Q: "I'm in a team and can't access a page I used to reach"**
> A: This should be resolved with this release. Ask the user to log out and back in. If the issue persists post-deployment, confirm their role assignment is correct and escalate.

> **Q: "A user keeps getting an Access Denied error on the Bills page"**
> A: Confirm the user has the correct role assigned. If their role should grant access, this was likely the bug fixed in this release — logging out and back in post-deployment should resolve it.

> **Q: "We were assigned to a team and now different things are visible"**
> A: Team assignment doesn't change your role — it just groups you with colleagues. Role permissions still apply as normal. This fix ensures they apply correctly.

### Supplier BPay Field Now Included in Bill Creation

When a new supplier is added from within the bill creation workflow, the BPay payment flag is now correctly sent.

**Why this was changed**

The form was missing the `is_bpay` field, meaning suppliers created this way could have had incorrect payment method records.

**What's different**

Previously: Creating a supplier via the bill workflow omitted the BPay indicator.

Now: The field defaults to false and is always included, ensuring accurate payment records.

**Where in the platform**

`Bills → Create Bill → Add Supplier` drawer.

**How to help users**

If users report that a supplier created via the bill workflow has incorrect payment type information, the record may need to be manually corrected for suppliers created before this fix.

**Likely Q&A**

> **Q: "A supplier I created from a bill doesn't show the right payment method"**
> A: If the supplier was created before this fix, the BPay field may have defaulted incorrectly. Ask the user to check and manually update the payment details on the supplier record.

> **Q: "Do we need to clean up any existing supplier records?"**
> A: Only if BPay suppliers were created via the bill workflow before this fix and the flag was expected to be set. Review any recently created suppliers if in doubt.

---

### MYOB Transaction Sync Changes (Background)

The MYOB transaction reconciliation job has been refactored. This is an internal background process change with no user-facing impact.

**Why this was changed**

The previous batch approach was susceptible to timeouts. Each sub-account is now processed independently for better reliability.

**What's different**

No visible change for end users. Transactions linked to funding records are preserved during re-syncs, preventing accidental data loss.

**Where in the platform**

Internal background job — not visible to users.

**How to help users**

If users report missing transactions after a sync, confirm whether they are linked to funding records (these are protected). Unlinked, non-pending transactions are deleted and re-fetched from MYOB on each sync.

**Likely Q&A**

> **Q: "Transactions are missing from a participant's ledger after a sync"**
> A: Transactions linked to funding records are protected. If unlinked transactions appear missing, they will be re-fetched on the next sync cycle. Escalate if data remains inconsistent after a full sync.

  ### Change Log

**[#6168](https://github.com/Trilogy-Care/tc-portal/pull/6168) — fix: fix permission issues with teams**

- New migration `2026_02_24_103800_fix_permission_table_primary_keys_for_teams`:
  - Converts existing NULL `team_id` values to `0` on `model_has_permissions` and `model_has_roles`
  - Alters both tables: `team_id` becomes `BIGINT UNSIGNED NOT NULL DEFAULT 0`
  - Drops and rebuilds composite primary keys to include `team_id` — new PKs: `(team_id, role_id, model_id, model_type)` and `(team_id, permission_id, model_id, model_type)`
  - Handles the case where `team_id` column doesn't exist yet (adds it)
- `app/Http/Middleware/CheckPermission` — `handle()` now calls `$request->user()->hasPermissionTo($ability)` directly instead of `$request->user()->currentRole->hasPermissionTo($ability)`
- `app/Http/Middleware/SetPermissionTeamContext` — added `else { setPermissionsTeamId(0); }` fallback so unauthenticated/teamless requests default to global scope (`team_id = 0`)
- `app/Jobs/ZohoSyncJobs/SyncJobs/SyncConsumerFromZohoJob` — added `setPermissionsTeamId(0)` at start of `handle()` to prevent team_id leaking from prior jobs in the same worker process
- `app/Providers/AuthServiceProvider` — added `setPermissionsTeamId(0)` in `boot()` to establish global default team context for CLI commands and queued jobs
- `domain/User/Models/User`:
  - Aliased `HasRoles::getAllPermissions` as `protected spatieGetAllPermissions` and `HasRoles::roles` as `protected spatieRoles`
  - New `roles(): BelongsToMany` override — returns all roles without team_id filter via direct `morphToMany` (fixes all `whereHas('roles', ...)` queries across the codebase)
  - New `permissions(): MorphToMany` override — returns all permissions without team_id filter
  - New `teamScopedRoles(): BelongsToMany` method — delegates to `spatieRoles()` for explicit team-scoped queries
  - Updated `hasPermissionTo()` — calls `spatieHasPermissionTo()` (now uses non-team-scoped `roles()`), falls back to `currentRole->hasPermissionTo()`
  - New `getAllPermissions(): Collection` override — delegates to `spatieGetAllPermissions()` via non-team-scoped relationships
  - Updated `syncPermissionsTeamContext()` — now calls `setPermissionsTeamId($this->current_team_id ?? 0)` (fallback to 0 instead of null)
- `domain/User/Seeders/UserSeeder` — admin user in dev seed now gets `Care Partner` and `Bill Processing Manager` roles, assigned to `Pod E1` and `Bill Processing` teams
- `tests/Feature/PermissionRegressionTest` — updated two test cases to use `setPermissionsTeamId(0)` instead of `null`
- `tests/Feature/TeamPermissionFixTest` — new 161-line test file covering: CheckPermission middleware with team context, `hasPermissionTo()` for global role permission with team set, direct permission with team set, Spatie context preservation, `getAllPermissions()` merge, role switching endpoint, super admin bypass, duplicate `assignRole()` idempotency, `whereHas('roles')` with active team context, middleware team context propagation

**[#6170](https://github.com/Trilogy-Care/tc-portal/pull/6170) — hotfix - SyncMyobTransactionsBySubAccountsJob with vc work and remove old GI fields** (DUC-184)

- New `Domain\Transaction\Jobs\SyncMyobTransactionsBySubAccountJob` (`ShouldBeUnique`, `ShouldQueue`, `public int $timeout = 180`):
  - `uniqueId()` returns `$subAccount` (prevents duplicate jobs per account)
  - Deletes transactions matching `myob_sub_account` that are NOT in `funding_transaction` table (preserves funding-linked records)
  - Calls `MyobService::fetchAllTransactions()` with `timeout: 170` (just under the job timeout)
  - Dispatches `VoluntaryContributionFundingUpsertAction` for `Transaction::VC_ACCOUNT` credit transactions with a matched package
  - Dispatches `MarkBillAsPaid` for bills in `BillStageEnum::PAYING` matching returned `RefNumber` values
  - Dispatches `UpdateTransactionWithSupplierReferenceNumberAction` for AP module transactions without a matched bill
- New `Domain\Transaction\Console\Commands\SyncMyobTransactionsBySubAccount` (command signature: `sync:myob-transactions-by-sub-account {sub_account}`, dispatches single `SyncMyobTransactionsBySubAccountJob`)
- `app/Console/Kernel` — replaced `SyncMyobTransactionsBySubAccounts::class` with `SyncMyobTransactionsBySubAccount::class` in the `$commands` array
- `Domain\Myob\Services\MyobService::fetchAllTransactions()` — added `?int $timeout = 120` parameter; HTTP client now passes `$timeout` instead of hardcoded `120`
- `Domain\Transaction\Jobs\SyncMyobTransactionsJob` — removed `$item['SubIDExt']` and `$item['AccountIDExt']` fallbacks from `Subaccount` and `Account` field access (removes old GI field support)
