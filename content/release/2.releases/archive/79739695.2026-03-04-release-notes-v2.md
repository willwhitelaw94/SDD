## TL;DR

- **Team permissions hotfix** — A critical issue where users on teams could not access features they had permission for has been fully resolved, including a migration to normalise team IDs across all permission tables.
- **Organisation codes and colours** — Every verified organisation is now automatically assigned a short 4-character code and a consistent colour for quick visual identification throughout the platform.
- **Budget V2 foundations** — The architectural foundations for the redesigned Budget module are in place, including a new funding streams API and refreshed Vue components for the budget planning experience.
- **Supplier dashboard document alerts** — Suppliers now correctly see expired and missing document warnings on their dashboard, even when they have no active services in their price list.
- **Billing visibility improvements** — Care Coordinators and suppliers can now see on-hold and rejection reasons directly on bills, and fee exemption tracking now records which user applied the exemption.
- **Support email updated** — Portal support queries now route to `portal.support@trilogycare.com.au` instead of the old IT inbox.

### Team Permissions Resolved

**Why this matters**

Team-based permission management is foundational to how organisations control what their staff can access. A bug introduced with the teams feature was causing access failures for users assigned to teams, undermining operational reliability and creating support overhead.

**What's changing**

The permission system has been fully repaired. Users assigned to teams now correctly inherit both their team-specific and global role permissions. The fix includes a database migration to normalise all existing permission records, ensuring the platform behaves consistently for all users regardless of whether they are in a team or not.

**Who's affected**

All organisations using team-based role management. Users in teams who reported unexpected access issues should find their access restored after this release.

**Action required**

No action required — the migration runs automatically on deployment. If any staff still report permission issues after the release, please contact support.

### Team Permissions Hotfix

**Why this was changed**

A bug in the team permissions implementation was causing users assigned to teams to be denied access to features their role should permit. This was causing support escalations and workflow disruption.

**What's different**

The `CheckPermission` middleware now calls `hasPermissionTo()` on the User model directly (which checks both team-scoped and global permissions), rather than through the current role only. A database migration normalises all `team_id` values from `NULL` to `0` across permission tables.

Old behaviour: Users in teams were sometimes denied access even with the correct role.

New behaviour: Permission checks consistently respect both team-scoped and global roles.

**Where in the platform**

Affects the entire portal — any permission-guarded route or action.

**How to help users**

If users report access issues after this release, first check their role assignment is correct, then confirm they are on the right team. The underlying bug is fixed — remaining issues are likely configuration.

**Likely Q&A**

> **Q: "I still cannot access a feature even though my role should allow it?"**
> A: First check the user's assigned role and team in the Admin panel. If the role is correct and they still cannot access it, please escalate with the user's email and the feature they are trying to access.

> **Q: "Did this change affect anyone's permissions?"**
> A: No permissions were removed. The fix only restores access that was incorrectly being blocked.
